## Почему слоты материализуются заранее
В ТЗ требуется, чтобы `slotId` был стабильным и сохранённым в БД. Поэтому выбран вариант материализации слотов в БД на скользящее окно `SLOT_HORIZON_DAYS`.

По умолчанию окно — 90 дней. Это сильно больше, чем типичный пользовательский сценарий из ТЗ (ближайшие 7 дней), и при этом остаётся дешёвым по объёму данных.

## Что делаем при сбоях Conference Service
Если пользователь запросил `createConferenceLink=true`, API сначала вызывает внешний mock service, а только потом пишет бронь в БД.

Поведение:
- если внешний сервис недоступен или вернул ошибку — бронь не создаётся
- если внешний сервис успел отдать ссылку, а запись в БД не сохранилась — возможна висячая ссылка без брони

## Валидация Auth
Для дополнительного задания с `/register` и `/login` реализована базовая валидация:
- нормализация email
- проверка email через стандартный пакет
- минимальная проверка длины пароля
- нормализация роли

## CI в GitHub Actions 
CI в GitHub Actions намеренно не включён в итоговый вариант

23.03.2026 было получено уточнение от организаторов задания: пункт про GitHub Actions рекомендуется не выполнять, чтобы не ограничить корректный запуск решения со стороны проверяющих.

По этой причине CI: classroom.yml и ci.yml, которые должны были находится в папке .github/workflows не включёны в итоговый сценарий сервиса
