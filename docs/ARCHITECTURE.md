# Архитектура
В этом файле подробно представлена информация об архитектуре сервиса

Сервис реализован как layered architecture:

- `handlers` — HTTP-слой, разбор запросов и формирование ответов
- `middleware` — JWT-аутентификация, проверка ролей, единый формат ошибок
- `service` — бизнес-логика
- `repo` — доступ к PostgreSQL
- `tx` — транзакционный менеджер
- `domain` — доменные сущности и enum-значения

## Основные сущности

- `Room` — переговорка
- `Schedule` — одноразово задаваемое недельное расписание переговорки
- `Slot` — материализованный 30-минутный слот
- `Booking` — бронь слота
- `User` — пользователь с ролью `admin` или `user`

## Почему слоты материализуются заранее

Самый нагруженный эндпоинт по ТЗ — получение доступных слотов по комнате и дате.  
Чтобы не вычислять слоты на лету при каждом запросе, после создания расписания сервис заранее генерирует слоты на горизонт вперёд и хранит их в таблице `slots`.

Это даёт:
- простой и быстрый `SELECT` для `/rooms/{roomId}/slots/list`
- предсказуемую нагрузку на БД
- возможность использовать индексы и простую фильтрацию по активным броням

## Как обеспечивается отсутствие двойного бронирования

В БД используется ограничение, которое запрещает более одной активной брони на один слот.  
За счёт этого даже при гонке параллельных запросов одновременно активной может остаться только одна бронь.

Дополнительно сервис маппит конфликт БД в бизнес-ошибку `SLOT_ALREADY_BOOKED`.

## Работа с транзакциями

Критические операции записи выполняются через `tx.Manager`:

- создание брони
- отмена брони
- другие write-сценарии, где важно атомарное изменение нескольких сущностей

Это позволяет не размазывать транзакционную логику по service-слою.

## Conference link

Если пользователь запрашивает `createConferenceLink=true`, бронь создаётся как основная операция, а ссылка на конференцию прикрепляется best-effort.

Такое поведение выбрано специально:
- бронь является первичной бизнес-сущностью
- временная недоступность внешнего сервиса конференций не должна приводить к потере уже созданной брони
- это снижает вероятность неоднозначного состояния для клиента

## Аутентификация и роли

Используется JWT.

Поддерживаются роли:
- `admin`
- `user`

Права:
- `admin` создаёт переговорки и расписания, смотрит общий список броней
- `user` бронирует слоты, смотрит свои брони и отменяет только свои брони

## Хранение времени

Все даты и время обрабатываются в UTC.  
Это соответствует спецификации и упрощает логику расписаний, слотов и броней.
